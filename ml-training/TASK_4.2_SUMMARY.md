## Technical Approach

### SageMaker Neo Compilation Process

1. **Input Model**:
   - Format: TensorFlow SavedModel
   - Source: Task 4.1 training output
   - Location: S3 bucket (vericrop-training-data-{account-id}/models/)

2. **Compilation Configuration**:
   ```python
   {
     'S3Uri': 's3://bucket/models/model.tar.gz',
     'DataInputConfig': '{"input_1":[1,224,224,3]}',
     'Framework': 'TENSORFLOW',
     'TargetDevice': 'rasp3b'  # ARM Cortex-A53
   }
   ```

3. **Optimization Techniques**:
   - Operator fusion (combine multiple ops)
   - Memory layout optimization
   - Quantization (FP32 → INT8 where possible)
   - Dead code elimination
   - Constant folding

4. **Output Model**:
   - Format: DLR (Deep Learning Runtime) compatible
   - Size: ~50% smaller than original
   - Inference: 2-3x faster
   - Location: S3 bucket (compiled-models/)

### Target Device: ARM Cortex-A53

**Why rasp3b target?**
- ARM Cortex-A53 architecture
- Similar to Android phones (Snapdragon 400/600 series)
- Compatible with AWS IoT Greengrass v2
- Widely available in Indian market

**Device Specifications**:
- CPU: ARM Cortex-A53 (quad-core, 1.2 GHz)
- RAM: 1-2 GB typical
- OS: Android 7+ or Linux (Greengrass)
- Network: 4G/LTE with intermittent connectivity

### Inference Latency Optimization

**Target: <2 seconds per inference**

**Latency Breakdown**:
1. Image loading: ~50ms
2. Preprocessing: ~100ms
3. Model inference: ~1200-1500ms (compiled)
4. Post-processing: ~50ms
5. **Total**: ~1400-1700ms ✓ Meets target

**Optimization Strategies**:
- Use compiled model (2-3x speedup)
- Batch preprocessing when possible
- Cache model in memory (avoid reload)
- Use efficient image libraries (Pillow, OpenCV)
- Minimize data copying

## Troubleshooting

### Compilation Fails

**Issue**: Compilation job fails with "Unsupported operator" error

**Solution**:
1. Check model uses only supported TensorFlow operations
2. Verify input shape configuration is correct
3. Try different target device (e.g., jetson_nano instead of rasp3b)
4. Check SageMaker Neo documentation for supported ops

### Latency Exceeds Target

**Issue**: Inference latency >2 seconds

**Solution**:
1. Verify using compiled model (not original)
2. Check device CPU/memory resources
3. Optimize preprocessing (use efficient libraries)
4. Consider model quantization (INT8)
5. Profile inference to identify bottlenecks

### Model Size Too Large

**Issue**: Compiled model still too large for edge device

**Solution**:
1. Use model pruning to remove unnecessary weights
2. Apply quantization (FP32 → INT8)
3. Consider smaller base model (MobileNetV3-Small)
4. Remove unused operations from graph
